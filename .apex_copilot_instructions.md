## 🏗️ Core Architectural Principles

### 1. Blueprint-First Architecture
- **Blueprints** are the single source of truth for test flows
- **Never store test scripts permanently** — always generate on-demand from blueprints
- Blueprint logic MUST be separated from execution logic

### 2. Multi-Agent System
- Agents are **separate micro-modules**
- Agents communicate **only through the agent runtime**
- Never merge agent logic into monolithic files
- Each agent has specialized responsibilities (QA, Security, Perf, DevOps, Docs)

### 3. AI Integration Layer
- Use the **LLM adapter layer** for all AI calls (LLM-agnostic design)
- AI assists in test generation, NOT in pass/fail decisions
- Keep AI logic separate from validation logic

### 4. Deterministic Validation Engine
- **NO LLM-based pass/fail decisions**
- Validation engine uses: blueprints, contracts, baselines, logs
- Bug detection is deterministic and evidence-based

### 5. Plugin Architecture
- Use **plugin loader** for external tool integration
- Tools are runtime-loaded, NOT hardcoded
- Support plug-and-play extensibility

### 6. Memory & Intelligence
- Use **vector memory** for test intelligence
- Learn from past test executions
- Store insights, not scripts

### 7. Multi-Application Isolation
- **Each application is a separate tenant** with isolated resources
- **Project Context** drives all agent operations
- **Never mix data** between applications
- Apps have isolated: blueprints, metadata, logs, snapshots, configs
- Agents operate on ONE app at a time (never globally)

## 🏛️ Multi-Application Architecture

### Application Isolation Structure
```
apps/
   app1/                    # ← Isolated tenant
      blueprints/           # App-specific test blueprints
      metadata/             # App-specific metadata
      logs/                 # App-specific execution logs
      snapshots/            # App-specific baseline snapshots
   app2/                    # ← Separate isolated tenant
      blueprints/
      metadata/
      logs/
      snapshots/
   app3/
      ...
```

**Critical Rules:**
- ✅ Each app is a **complete isolated tenant**
- ✅ Nothing overlaps between apps
- ✅ Each app has its own environment configs
- ✅ Each app can use different plugins
- ✅ Each app can use different LLM models
- ❌ Agents NEVER operate globally
- ❌ Never query across app boundaries
- ❌ Never share blueprints between apps

### Project Context Pattern

Every agent operation receives a **Project Context** object:

```python
@dataclass
class ProjectContext:
    """Context for isolated application operations"""
    app_id: str                    # e.g., "app1"
    blueprint_path: Path           # /apps/app1/blueprints
    env_config: Path               # /environments/qa/app1.yaml
    snapshots_path: Path           # /apps/app1/snapshots
    logs_path: Path                # /apps/app1/logs
    metadata_path: Path            # /apps/app1/metadata
    plugins: List[str]             # ["playwright", "zap", "jmeter"]
    llm_model: str                 # "qwen-7b" or "gpt-4"
    memory_path: Path              # /core/memory/app1.db
    environment: str               # "dev", "qa", "staging", "prod-safe"

# ✅ CORRECT: Agent receives context
class QAAgent:
    def generate_test(self, blueprint: Blueprint, context: ProjectContext):
        # All operations scoped to context.app_id
        blueprints = self.load_blueprints(context.blueprint_path)
        snapshots = self.load_snapshots(context.snapshots_path)
        # ...

# ❌ WRONG: Agent operates globally
class QAAgent:
    def generate_test(self, blueprint: Blueprint):
        # No context! Which app is this for?
        blueprints = self.load_blueprints("/blueprints")  # WRONG!
```

### How Agents Distinguish Applications

```python
# ✅ CORRECT: Context-aware agent operation
def run_qa_tests(app_id: str):
    # Build context for specific app
    context = ProjectContext(
        app_id=app_id,
        blueprint_path=Path(f"/apps/{app_id}/blueprints"),
        snapshots_path=Path(f"/apps/{app_id}/snapshots"),
        # ...
    )
    
    # Agent operates within context boundaries
    agent = QAAgent()
    results = agent.execute_tests(context)
    
    # Results stored in app-specific location
    save_results(context.logs_path, results)

# ❌ WRONG: Cross-app contamination
def run_qa_tests():
    # No app_id! Operating globally!
    blueprints = load_all_blueprints()  # Mixes app1, app2, app3!
    agent = QAAgent()
    results = agent.execute_tests(blueprints)  # Which app?
```

### Multi-App Code Generation Rules

**✅ DO:**
- Always require `ProjectContext` in agent methods
- Use `context.app_id` to scope all file operations
- Load blueprints from `context.blueprint_path`
- Save logs to `context.logs_path`
- Use `context.plugins` to load tools
- Use `context.llm_model` for AI operations

**❌ DON'T:**
- ❌ Don't use global paths (`/blueprints`, `/logs`)
- ❌ Don't query across apps
- ❌ Don't share memory between apps
- ❌ Don't mix app1 blueprints with app2 execution
- ❌ Don't create global singletons for app-specific data

## 📖 Example Code Style

### ✅ GOOD - Blueprint-First, Modular:
```python
# agents/qa_agent.py
class QAAgent:
    def __init__(self, blueprint_loader, test_generator, validator):
        self.blueprint_loader = blueprint_loader
        self.test_generator = test_generator
        self.validator = validator
    
    def execute_test(self, blueprint_id):
        blueprint = self.blueprint_loader.load(blueprint_id)
        test = self.test_generator.generate_from_blueprint(blueprint)
        result = test.execute()
        return self.validator.validate(result, blueprint)
```

### ❌ BAD - Script Storage, Monolithic:
```python
# test_runner.py (3000 lines)
def run_all_tests():
    # Hardcoded test scripts
```

## 🔄 Typical Workflow

1. User defines test flow via **Web UI** (codeless)
2. System saves definition as **blueprint** (not script)
3. Agent runtime loads appropriate **specialized agent**
4. Agent uses **LLM adapter** to generate test from blueprint
5. Test executes in **sandbox environment**
6. Results collected and sent to **validation engine**
7. Validator checks against **blueprint + contracts + baselines + logs**
8. Deterministic **pass/fail decision** made
9. If fail, **auto-healing** regenerates test
10. Results stored in **observability** system
11. Insights saved to **vector memory**
